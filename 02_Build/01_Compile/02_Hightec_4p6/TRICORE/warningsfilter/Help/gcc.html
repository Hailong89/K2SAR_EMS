See also 12488 . Make sure that left and right sides in assignments, casts, initializers and function arguments are of the same type and no qualifiers like volatile or const etc. are discarded. These keywords are not just decorations but may be essential for generating correct and efficient code. For example the compiler may use more optimizations which e.g. may remove memory accesses thus changing the functional behaviour of your program. The following code discards some qualifiers and will issue warnings of that kind: 
void foo (int *arg1);
volatile int i;
const int c = 0;

int
main(void)
 {
	 int *pc;
	 ...
	 foo (&i);
	 pc = &c;
 }
To avoid the warning change the code like that: 
void foo (volatile int *arg1);
volatile int i;
const int c = 0;

int
main(void)
 {
	 const int *pc;
	 foo (&i);
	 pc = &c;
 }

10375 comparison between signed and unsigned
While comparing integer based variables (that are e.g. char, short, int) the variables are implicitly casted to the same type. This is done using the integer promotion: if the signed version of the new type can hold all valid values of the original type the implicit cast is done to the signed type, otherwise the unsigned type is used. In the following example the short variable v2 is implicitly casted to a signed int because signed int variables can hold all valid values of a singed short. A code sequence like 
int
main (void)
{
        unsigned int v1;
        short v2;

        if (v1 > (unsigned short)v2 << 8)
         {
          /* code */
         }
        return 1;
}
will issue this warning if the option -W is set. In the example the variable v2 is casted as unsigned, but because of the shift operation the result of the value is implicitly converted to an signed int value. Therefore the warning is generated. If you are using operand like +,-,~,<<,>> you should cast the result of the operation like (unsigned int)((unsigned short)v2 << 8)). ISO C99 (chapter 6.3.1 on page 42-43 and footnote 48) says: The integer promotions are applied only: as part of the usual arithmetic conversions, to certain argument expressions, to the operands of the unary +, -, and ~ operators, and to both operands of the shift operators, as specified by their respective subclauses.

For C (and C++), gcc makes optimizations based on the type of expressions. In particular, an object of one type is assumed never to reside at the same address as an object of a different type, unless the types are almost the same. For example, an unsigned int can alias an int, but not a void* or a double. A character type may alias any other type. Pay special attention to code like this: 
union a_union
{
	int i;
	double d;
};

int f() 
{
	a_union t;
	t.d = 3.0;
	return t.i;
}
The practice of reading from a different union member than the one most recently written to (called 'type-punning') is common. Even with -fstrict-aliasing, type-punning is allowed, provided the memory is accessed through the union type. So, the code above will work as expected. However, this code might not: 
int f() 
{
	a_union t;
	int* ip;
	t.d = 3.0;
	ip = &t.i;

	return *ip;
}

Compound statements is code sourrounded by curly braces, e.g. code that is part of an if-statement. A compound statement may include labels. But labels must be assigned to valid C/C++-Code. If a label is placed at the end of compound statement, there is no C/C++-Code the compiler can assign to the label and therefore this warning is issued. For this code the compiler will output the warning: 
if (a == b)
 {
	x = 1;
	label1:
  }
If you need to have the label at the end of the compound statement you must add a null instruction after the label: 
if (a == b){
	x = 1;
	label1: ;
}
A common case in which this warning may occur is the switch-case-instruction. Here you have probably forgot to add code to the last case or default instruction. The case and default instructions are regarded as labels by the compiler. For this code the compiler will output the warning:

	case 0:
	   b = 1;
	   break;
	case 1:
	   b = 0;
	   break;
	default:
 }

switch(a)
 {
	case 0:
	   b = 1;
	   break;
	case 1:
	   b = 0;
	   break;
	default:
	   break;
 }
